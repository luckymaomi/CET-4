# kaoshi 考试生命周期和写作题阅卷闭环计划

## 需求

- 把考试草稿、发布、撤销发布、关闭、删除、更新试卷和阅卷生命周期做到生产级标准。
- 草稿和发布是同一个考试业务对象的状态变化，不能做成两套可分离业务对象。
- 发布必须生成不可变发布快照；作答必须基于发布快照生成 attempt 快照；成绩必须基于 attempt 快照可追溯。
- 全链路必须无魔法：题库试题编辑不隐式影响考试草稿和已发布快照；考试草稿只有保存草稿才持久化；试卷只有点击更新试卷才按题库重新生成；发布只发布当前已保存草稿题目明细。
- 删除考试时必须按生命周期保护数据：没有作答/成绩时可物理删除整个考试及草稿、发布快照；已有作答或成绩时不能硬删除。
- 增加写作题 `WRITING`，并做到题库、试卷、发布快照、作答快照、作答保存、提交、人工阅卷保存、完成阅卷、最终成绩归档的生产级闭环。
- 阅卷必须支持中途保存、退出、重新进入继续阅卷；只有显式完成阅卷后，成绩才进入最终状态。
- CET4 真题迁移本轮先不做，等生命周期和写作题闭环稳定后再讨论迁移策略。

## 当前事实

- 当前 `exams.status` 已表达 `DRAFT`、`PUBLISHED`、`CLOSED`，考试本体是同一个对象。
- 当前发布时会从 `exam_draft_questions` 生成 `exam_published_questions`，撤销发布会删除发布快照并把考试改回 `DRAFT`。
- 当前显式更新试卷已存在，但需要用自动化测试保护“题库变化不魔法影响试卷，点击更新试卷才更新”。
- 当前没有删除考试接口和前端删除入口。
- 当前题型只有 `SINGLE_CHOICE` 和 `MULTIPLE_CHOICE`。
- 当前题库试题保存要求至少两个选项，Excel 导入也要求选项和正确答案。
- 当前考试规则只支持单选题数量/分值和多选题数量/分值。
- 当前考试端答案只保存 `selectedLabels`，没有主观题文本答案。
- 当前提交会自动评分所有题目，成绩没有待阅卷状态。
- 当前成绩详情只展示选择题选项、我的答案、正确答案和正确/错误。
- `C:\Users\Administrator\Desktop\exam-platform-research\online-examination-platform` 的成熟参考项目包含在线考试自动保存、客观题自动评分、主观题人工评分、成绩分析和“待批改”展示；本项目吸收这些生命周期原则，但不引入其微服务复杂度。

## 生产级验收保护点

- 草稿、发布、关闭是同一考试对象的状态；发布快照和作答快照是历史事实，不作为独立可删除业务对象暴露给用户。
- `DRAFT` 可编辑、发布、删除；删除会清理考试本体、规则、开放部门、草稿题目明细。
- `PUBLISHED` 且无作答/成绩时可撤销发布回草稿，也可删除；删除会同时清理草稿和发布快照。
- 一旦存在 attempt 或 result，考试不能撤销发布，不能物理删除；只能关闭，保留发布快照、作答快照、答案、阅卷和成绩。
- 保存草稿后重新打开必须读回完整草稿：基础配置、开放部门、组卷规则、题目明细、顺序、分值和题型。
- 题库试题编辑后，既不隐式改写已保存草稿，也不隐式改写发布快照；只有点击更新试卷并保存草稿后，考试草稿题目明细才更新。
- 发布考试只发布当前已保存草稿题目明细；发布后再编辑考试必须回到草稿编辑语义，再显式发布新快照。
- 写作题不是选择题变体，不保存选项和客观正确答案。
- 写作题必须进入草稿题目明细、发布快照和作答快照，题干、解析/参考、分值、顺序和附件都要快照化。
- 考试端写作题必须用文本框作答，支持自动保存、手动提交前保存、刷新读回、提交锁定和结果详情读回。
- 作答保存语义必须显式：作答过程使用短延迟防抖自动保存；同一题保存必须串行，后一次答案不能被前一次慢请求覆盖；提交前必须整卷 flush 当前前端状态并等待成功后再提交锁定；刷新读回只验证已确认保存成功的答案；浏览器关闭/刷新不能作为强一致保存点，只能在存在未保存变更时拦截提示。
- 提交含写作题的试卷后，选择题自动评分，写作题进入待阅卷；成绩列表和详情必须明确显示待阅卷，不能把未阅卷成绩伪装成最终成绩。
- 阅卷保存是可中断动作：保存某一道写作题分数和评语后，关闭页面再打开必须读回。
- 阅卷完成是显式动作：所有写作题均已保存分数后，管理员点击完成阅卷，成绩才进入完整归档：总分、客观题分、主观题分、通过状态、阅卷人、阅卷完成时间都可追溯。
- 已完成阅卷的成绩默认锁定，不能静默覆盖；后续如需重评必须新增“重评/更正”流程，本轮不做。
- 题型定义必须集中管理，后端和前端不散落单选/多选/写作硬编码判断；新增题型时优先扩展题型元数据和明确策略。
- 文件职责必须按变化原因审查：超过 300 行必须触发职责审查；状态管理、规则计算、数据读写、渲染展示、外部接线、错误兼容、业务判断混在一起时，当前功能收口后必须进入下一步拆分计划。
- 浏览器自动化必须验证：草稿保存后重新打开读回，编辑后再次保存读回；题库改题不影响已有草稿，点击更新试卷后才更新；发布后考生看到发布快照；写作答案保存后刷新读回；阅卷保存后退出再进入读回；完成阅卷后成绩固定。
- 后端测试、前端类型检查/单测、真实浏览器验收必须覆盖新生命周期和写作题链路。

## 设计

- 考试生命周期：
  - 新增 `DELETE /api/admin/exams/{id}`。
  - 删除前检查 `exam_attempts` 和 `exam_results`；存在任一记录则返回冲突。
  - 可删除考试按外键顺序清理发布附件/选项/题目、草稿附件/选项/题目、规则、开放部门和考试本体。
  - 撤销发布继续只允许无作答/成绩的已发布考试。
  - E2E 明确验证草稿保存读回、题库改题不魔法更新、显式更新试卷、发布快照稳定。

- 题型元数据：
  - 后端新增集中题型定义，至少覆盖 `SINGLE_CHOICE`、`MULTIPLE_CHOICE`、`WRITING` 的名称、是否选择题、是否需要选项、是否自动评分、是否人工阅卷。
  - 前端新增同等题型元数据，页面只通过元数据判断展示方式、选项需求、答案列和题型文本。

- 写作题模型：
  - 题型新增 `WRITING`。
  - `question_options` 对写作题为空；后端校验写作题不得带选项。
  - 题库统计增加 `writingCount`。
  - 试题 Excel 的写作题允许选项和正确答案为空。

- 组卷规则：
  - `exam_rules` 增加 `writing_count` 和 `writing_score`。
  - 规则组卷按题型元数据定义的顺序生成草稿题目明细。
  - 手工选题沿用统一题目明细，写作题默认分值从规则或默认值读取。

- 作答与评分：
  - API 请求同时支持 `selectedLabels` 和 `answerText`。
  - `exam_answers` 增加 `answer_text`、`review_comment`、`reviewed_by`、`reviewed_at`，选择答案字段允许为空，`is_correct` 允许为空。
  - 选择题提交时自动评分；写作题提交时保存文本、分数为 0、`is_correct = null`。
  - `exam_results` 增加 `objective_score`、`subjective_score`、`grading_status`、`reviewed_at`。
  - 含未完成阅卷写作题的成绩为 `PENDING_REVIEW`；完成阅卷后变为 `FINAL`。

- 作答保存：
  - 外部资料确认 `beforeunload`/页面关闭链路不能承载可靠异步保存，`sendBeacon` 也只适合少量尽力发送数据；成熟考试项目更常见的是整卷作答记录加单题答案明细，而不是把 UI 点击当成最终事实。
  - 当前主干重建为整卷答案快照保存：`/answers` 接收并保存当前 attempt 的所有题目答案，不保留单题保存兼容语义。
  - 后端按 attempt 快照题目清单校验答案快照：每道题必须出现且只出现一次，题目不属于本次作答或重复提交都拒绝。
  - 前端采用防抖自动保存而不是每个输入事件立即请求；300ms 用于合并连续输入和多选连续点击，但每次触发保存的 payload 都是整卷当前状态。
  - 自动保存、路由离开前保存、提交前保存都走同一个整卷保存队列；旧请求完成时不得清掉新版本的待保存状态，最终落库必须是最新整卷状态。
  - 路由离开前先尝试 flush 当前答案；仍失败时给出明确风险确认。
  - 浏览器刷新/关闭时只在存在未保存或正在保存答案时触发浏览器原生拦截；不承诺在关闭过程中完成异步保存。
  - 提交和自动交卷是强一致点：先整卷保存当前状态，保存成功后才调用提交接口，提交后 attempt 锁定。

- 阅卷：
  - 新增管理员接口保存单个成绩中的写作题评分，保存后不自动完成整卷。
  - 新增管理员接口完成整份成绩阅卷，要求所有写作题已评分。
  - 后端校验题目属于该成绩、题型为写作题、分数在 `[0, 题目分值]`。
  - 每次保存阅卷后重算当前草稿成绩：客观题分、主观题已评分分、正确题数；状态仍可为 `PENDING_REVIEW`。
  - 完成阅卷后重算最终成绩：总分、客观题分、主观题分、正确题数、阅卷状态、通过状态和阅卷完成时间。
  - 成绩详情展示写作答案、得分、评语、保存状态、待阅卷/已阅卷/已完成状态。

## 下一步职责拆分/边界治理计划

当前生命周期、写作题闭环和整卷答案快照保存已经通过后端、前端和真实浏览器验收。下一步单独开任务做职责拆分，不混入本轮功能交付。

### 需求

- 治理超过 300 行且变化原因混杂的文件。
- 不改变当前产品行为、接口语义和数据库事实，只拆清边界、提升可测试性和后续维护性。

### 当前事实

- `ExamService.java` 承载考试 CRUD、草稿题目、发布快照、作答快照、整卷答案保存、提交评分、阅卷、结果详情和 Excel 导出，变化原因已经混杂。
- `ExamsView.vue` 承载列表、弹窗、规则组卷、手工选题、草稿保存、发布、删除、成绩抽屉和多种表单规则，页面状态、规则计算和 API 接线混在一起。
- `QuestionBanksView.vue` 承载题库树、分类、题库、试题编辑、附件和 Excel 导入导出，后续维护成本高。
- `ExamSessionView.vue` 已把作答保存生命周期抽到 `useAnswerSnapshotSaver`，但题目渲染、答题卡、倒计时和作答状态仍在一个页面里。
- `ExamResultReview.vue` 同时承载考生只读复盘和管理员阅卷编辑，边界可以进一步拆清。

### 生产级验收保护点

- 拆分后所有接口路径、请求响应、错误码、权限边界和页面用户路径保持当前事实。
- 整卷答案快照保存仍然是唯一作答保存语义；不能恢复单题保存兼容入口。
- 草稿、发布、删除、显式更新试卷、写作阅卷保存和完成阅卷的 E2E 行为必须全部保持。
- 每次拆分必须有对应测试或现有测试覆盖，不能只做文件搬家。

### 目标

- 后端把 `ExamService` 拆成清晰协作边界：考试生命周期、试卷草稿/发布快照、作答 attempt/答案快照、评分/阅卷、导出下载。
- 前端把考试管理大页面拆成业务规则模块、API 编排函数和展示组件，页面只保留路由级状态和用户交互编排。
- 考试端继续收敛：题目作答组件、答题卡组件、倒计时/提交逻辑、整卷保存 composable 分离。
- 结果详情拆出只读复盘和管理员阅卷表单，避免考生视图与管理编辑规则互相影响。

### 不做范围

- 不新增题型、不迁移 CET4 真题、不调整数据库结构。
- 不改变当前 UI 信息架构和按钮文案，除非拆分时发现明确错误。
- 不做历史兼容层。

### 设计

- 后端优先按业务流程变化原因拆分，不按行数机械拆分。
- Mapper 可先保留集中接口；只有当 SQL 变化原因也明显混杂时，再拆 Mapper。
- 前端优先提取纯规则和 payload 构造，再拆展示组件；先保障测试，再移动 UI。
- 每个新模块必须能一句话说明职责，不能形成新的杂物间。
