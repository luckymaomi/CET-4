# kaoshi Agent 工作规约

本文件是 kaoshi 仓库的最高协作约束。产品事实看 `spec.md`，中大型任务的执行合同按需写入 `plan.md`。

## 基础规则

- 始终使用简体中文和项目 owner 交流。
- 阅读本仓库文本文件时按 UTF-8 读取。
- 在 PowerShell 中运行 Node/npm 命令时使用 `npm.cmd`、`npx.cmd`。
- 不回滚 owner 已有改动，除非 owner 明确要求。
- 遇到脏工作区，只处理当前任务相关文件。
- 每次 commit 或 push 前必须先向 owner 确认。
- 不使用阶段性版本命名组织产品事实、计划或回复。

## 快速演进规则

kaoshi 是快速演进项目。当前生产级目标高于历史形态。

- 代码、接口、页面、数据结构、测试和文档以当前事实为准。
- 需要重建时直接删除旧表达、旧路径、旧接口、旧页面、旧数据形态并建立当前主干。
- 数据库脚本表达当前初始化事实；数据结构变化时重建初始化脚本和测试数据，验收以清空数据库后重新初始化为准。
- 大量删除、移动、重写和新建代码是允许的，判断标准是当前主干是否清晰、可测、可维护。
- 历史数据保留、旧接口保留、旧页面保留和兼容过渡只在 owner 明确要求并写入当次 `plan.md` 后执行。

## 根文档职责

- `AGENTS.md`：agent 工作规则和仓库协作纪律。
- `spec.md`：当前产品事实、业务边界、核心概念和验收标准。
- `plan.md`：当前任务的单文件规格驱动执行合同；按任务需要创建、重写、维护和收口。
- `plan.example.md`：后续任务计划模板。
- `README.md`：面向新读者的项目入口和产品介绍。
- `LICENSE`：开源协议权威文本。
- `SECURITY.md`：安全报告和敏感数据处理规则。
- `CONTRIBUTING.md`：贡献流程和验证规则。
- `.gitignore`：不进入版本控制的文件边界。
- `.codex/skills/kaoshi-development/SKILL.md`：开发、后端、前端、数据模型、测试和生产级交付规则。
- `.codex/skills/kaoshi-ui/SKILL.md`：管理端和考试端 UI 规则。
- `.codex/skills/plan/SKILL.md`：`plan.md` 写作和执行规则。

## Skills

命中 skill 描述时，必须先读取对应 `SKILL.md`，再行动。

- 修改 Java、Spring Boot、MySQL、Redis、Vue、TypeScript、接口契约、数据模型、测试、README、spec、AGENTS、skills 或运行配置时，先读 `.codex/skills/kaoshi-development/SKILL.md`。
- 修改页面信息架构、管理端表单、考试端作答页、HTML/CSS、Element Plus 组件、布局、按钮、空状态、表格或视觉样式时，先读 `.codex/skills/kaoshi-ui/SKILL.md`。
- 中大型开发、跨模块重构、生产级验收、架构硬化、评测闭环，或任何不能靠一次小补丁完成的任务，先读 `.codex/skills/plan/SKILL.md`，并按任务需要维护根目录 `plan.md`。

## 事实纪律

- 事实高于一切。
- 所有判断、计划、文档修改和回复必须基于当前仓库文件、命令输出、owner 明确提供的信息或可复现结果。
- 必须区分已确认事实、owner 提供事实、未验证推测、计划事项和已完成并验证事项。
- 文档只写当前事实和当前边界。
- 不把未实现能力写成当前产品事实。
- 不用口头解释替代测试结果和命令输出。

## 文件职责

- 单一职责看变化原因，不看行数。
- 超过 300 行必须触发职责审查，但不是自动拆分理由。
- 能一句话说清职责、变化原因一致、内部耦合合理，可以保留。
- 状态管理、规则计算、数据读写、渲染展示、外部接线、错误兼容、业务判断混在一起时，必须拆清边界。
- 不为了拆而拆，不为了省事硬塞。

## Plan 规则

`plan.md` 是单文件规格驱动执行合同。中大型任务、跨文档修改、产品边界调整、工程实现和验证闭环都必须先创建或维护 `plan.md`。

执行规则：

- 写完计划，再动手。
- checklist 必须随进度更新。
- 新事实推翻计划，先改计划，再继续。
- 不偷偷偏离计划。
- research、设计、实现、测试、文档同步、验证和收口做成同一次完整交付。
- 分阶段只表示交付边界，不表示降低标准；当期范围必须按生产级标准完成。

`plan.md` 必须覆盖：

- 需求。
- 当前事实。
- 生产级验收保护点。
- 目标。
- 不做范围。
- 设计。
- 实施任务。
- 验证计划。
- 收口。

## 产品主线

当前产品名是 kaoshi。

kaoshi 是现代化考试与考试管理平台。核心业务围绕用户、角色、权限、部门、题库、试题、考试、发布快照、作答快照、评分、成绩和文件附件展开。

## 技术纪律

- 后端使用 Java 21、Spring Boot 3、MySQL 8、MyBatis-Plus、Spring Security、JWT 和 Redis。
- 前端使用 Vue 3、TypeScript、Vite 和 Element Plus。
- 领域模型必须有显式类型。
- 后端接口必须有请求、响应、错误码和鉴权边界。
- 数据库变更必须落到当前初始化/重建脚本。
- 页面文件只负责交互和呈现，不承载复杂业务规则。
- 业务规则放在可测试模块中。
- Controller 负责接入，Service 负责业务流程，Mapper 负责数据访问，Excel/文件解析放入专门工具或服务，Vue 页面负责交互编排和展示。
- 只有当拆分能降低真实复杂度、隔离变化、提升测试性或匹配现有架构边界时，才新增模块或抽象。
- 禁止用 `any` 或无类型对象逃避设计。

## 测试纪律

- 测试保护当前核心能力：登录、首次改密、权限、用户导入导出、题库试题、考试保存读回、发布快照、作答快照、提交锁定、自动评分、成绩详情和真实浏览器链路。
- 后端测试保护接口契约、业务规则、生命周期、Excel 导入导出和权限边界。
- 前端单元测试保护 API client、状态流和独立前端规则。
- Playwright 测试保护真实用户路径和真实前后端联通。
- 单元测试可以隔离依赖；端到端验收必须使用真实后端、真实前端和真实数据库。
- 不能把没有真实运行的链路写成已通过。

## 工程入口

本地开发入口：

- `python .\start_dev.py`：启动 Docker MySQL/Redis、Java 后端、Vue 前端，并打开浏览器。
- `python .\start_dev.py` 默认清空本项目 Docker 数据库后启动。
- `python .\stop_dev.py`：停止本项目本地服务。

本地测试入口：

- `python .\start_test.py`：运行常规验证，不打开浏览器。
- `python .\start_browser_test.py`：重置本项目 Docker 数据库，启动真实后端和前端，运行有头 Chromium 浏览器验收。
- `python .\stop_test.py`：停止测试相关服务。

底层脚本：

- `scripts/verify.ps1`：Python 脚本编译、后端测试、前端类型检查、前端单元测试、前端构建、Docker Compose 配置和文档扫描。
- `scripts/run-e2e-stack.ps1`：真实浏览器验收的服务栈启动脚本。

## 验证规则

- 文档任务必须检查目标文件是否存在。
- 代码任务收尾前必须运行项目定义的类型检查、测试、构建或等价验证。
- 生产级收口必须优先运行 `python .\start_test.py` 和 `python .\start_browser_test.py`。
- 不能只凭解释声称完成，必须说明验证结果。
- commit 或 push 前必须说明将提交的范围并得到 owner 明确确认。

## 收尾回复

- 最终回复保持简洁。
- 说明改了什么、验证了什么、是否还有风险。
- 不粘贴大段日志、diff 或源码，除非 owner 明确要求。
